Visualisation Component

For this project, I propose to use the MPAndroidChart library, since it is well documented, provides a variety of useful features and extensive customization, a lot of different chart types and is easy to use without the need for much setup. In fact, I have used this library in one of my previous projects (see FinBuddy) and would very much lean towards using it again.

Since this is my main project of interest, I have included a Timeline section dedicated for the project. In terms of workflow, my plan for the project would be:

  1. Install the MPAndroidChart dependency to the build
  2. Create an abstract class ChartComponent which extends from AndroidViewComponent for charts (since charts share a lot of common attributes, events and methods). Since all of the charts have their own counterparts (such as LineDataSet for the DataSet of the LineChart), I will most likely go with making the ChartComponent abstract class a generic class, so we can pass in the class type per chart.
  3. Establish and make a UML diagram of the chart attributes. This way, I will be able to see the common functionality between charts, so that it can all be added to the abstract class. For instance, each chart has a Legend, DataSet, XAxis, YAxis, Colors, ValueFormatter, AxisValueFormatter and various style settings. Therefore methods setting and getting these values can be added in the ChartComponent class.
  4. Following up on the UML diagram, concrete charts have their own specific functionality. For instance, PieChart has an attribute whether to show the slice labels in percentage or as values. As such, these will be included in the UML diagram.
  5. Moving on to the implementation, after the UML diagram is fully established, I can then proceed to implementing the actual attributes and relevant methods with annotations for each chart, starting with ChartComponent.
  6. After ChartComponent is done, I will then proceed with concrete chart implementations. I plan to implement every chart that is available in the MPAndroidChart library, and for each chart, I will make a seperate class that extends the abstract ChartComponent class. The components will then be put under a new category "Charts". In the end, the following classes for concrete implementations will be present: LineChart, BarChart, PieChart, ScatterChart, BubbleChart, RadarChart, CandleStickChart. (each one is shown in the following section)
  7. After we are done with the attributes, events are needed for added functionality to the charts. Fortunately, MPAndroidChart provides various Gestures and an OnGestureListener which works via callbacks and MotionEvents. The gestures include the typical Android view events (long-press, tap, double-tap, touch up, touch down) plus a few extra ones related to the charts (zoom, drag). I plan to implement all of these. In general, these apply to every chart and will go to ChartComponent.
  8. Create an abstract class MockChartComponent that extends from MockVisibleComponent (so that we may see our visual changes in the editor itself)
  9. [OPTIONAL] Write unit tests for each Chart component. (This is part of another project, therefore it might not be appropriate here. However, in any case, I will make sure to test the Chart Components as thoroughly as possible)
  10. For each concrete chart class, create a class for the Mock Component that extends from MockChartComponent.
  11. Test each component in different browsers (just to be sure it works)
  12. Write the corresponding documentation for each Chart component, possibly under a new page of category Charts.
  13. [BONUS] MPAndroidChart provides the ability to construct Custom Markers when clicked on an entry on a Chart. This can be well put to use in this project to allow even more customization to the charts or have some sort of standard for the markers. Therefore I propose this as a bonus feature for the project. See link for reference.

Menu Options

General summary of the plan for the adding Menu Options project:

  1. Create new component MenuOptions that extends the AndroidNonvisibleComponent class. This could then be put under the UserInterface category.
  2. Create getters and setters for the property "Menu Elements" which would represent all user defined Menu Elements. I believe it would be a good idea to keep the current default menu options remain there, but perhaps keep them at the end of the list.
  3. Write additional code in the component itself to handle adding the Menu Elements into the context menu and attaching listeners to them.
  4. Make all menu options use the same listener, but make the event return a certain value (either Menu entry name or index) through dispatchEvent.
  5. Write an event for the listener, which would have one attribute - the value returned by the callback (Menu entry name/index)
  6. Additional alterations to Form.java might be needed, since two menu options are added in that class.
  7. Write documentation for the component.
